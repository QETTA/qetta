> **작성일**: 2026-02-01
> **버전**: 2.0
> **프로젝트**: KidsMap — 유아 발달 데이터 추적 앱 (QETTA와 독립된 별도 프로젝트)

## 0. 프로젝트 정체성

```
KidsMap ≠ QETTA

- KidsMap: 유아 발달 데이터 추적·시각화 앱 (독립 프로젝트)
- QETTA: AI 기반 정부지원사업 자동화 플랫폼 (별도 프로젝트)
- 두 프로젝트는 기술 스택을 공유할 수 있으나, 도메인·사용자·비즈니스 모델이 완전히 다름
```

```markdown
# CONTEXT

## 프로젝트 정체성
- 프로젝트명: KidsMap
- 도메인: 유아 발달 추적 (영유아 0~7세)
- 독립 프로젝트 (QETTA와 별개)
- 디자인 리소스: Tailwind Catalyst UI Kit, Compass 템플릿

## 대상 사용자
- 1차: 부모 (아이의 성장·발달 기록 및 시각화)
- 2차: 유치원/어린이집 교사 (반별 발달 관리)
- 3차: 아동 발달 전문가 (상담·치료 데이터 활용)

## 유아 데이터 범위
- 신체 발달: 키, 몸무게, 대근육, 소근육, 성장 곡선

- 언어 발달: 어휘량, 문장 길이, 발음 명료도
- 사회·정서: 또래 관계, 감정 조절, 자기 표현, 분리 불안
- 생활 습관: 수면 패턴, 식사, 배변 훈련, 자조 기술
- 건강 기록: 예방 접종, 알레르기, 질병 이력

# EXPLORATION AXES (탐색 6축)

## 축 1: 현재 상태 진단 (AS-IS)

다음을 파악하라:

### 1.1 기능 인벤토리
- 현재 구현된 기능 목록 (완성도 %와 함께)
- 사용 중인 UI 프레임워크/라이브러리
- 기존 데이터 모델 (유아 프로필, 발달 기록 등)

### 1.2 사용자 피드백 (있다면)
- 부모 사용자의 주요 불편 사항
- 가장 요청이 많은 기능
- 이탈 포인트 (어디서 사용을 중단하는가)

### 1.3 기술 현황
- 프론트엔드 스택
- 백엔드/API 구조
- DB 스키마
- 상태 관리
- 디자인 리소스 현황 (Catalyst, Compass 등)

---

## 축 2: 목표 상태 정의 (TO-BE)

### 2.1 비전
- 6개월 후 KidsMap의 이상적 상태
- 핵심 KPI (MAU, 기록 빈도, 리텐션, 공유율)
- 수익화 모델 방향 (프리미엄, B2B, 광고 등)

### 2.2 핵심 사용자 여정
```
부모 여정:
아이 등록 → 발달 기록 입력 → 시각화 대시보드 확인
→ AI 발달 인사이트 → 전문가 상담 연결 → 보고서 출력

교사 여정:
반 등록 → 원아별 발달 관찰 기록 → 반별 통계
→ 학부모 공유 → 발달 보고서 출력 (학기별)
```

### 2.3 기술 목표
- 성능: FCP < 1s, LCP < 2.5s
- 접근성: WCAG AA (부모 포함 다양한 연령대 사용)
- 오프라인: 기록 입력은 오프라인에서도 가능
- 보안: 유아 개인정보 보호 (아래 축 6 참조)

---

## 축 3: 갭 분석 (GAP)

AS-IS와 TO-BE 사이의 갭을 분류:

| 카테고리 | 갭 항목 | 심각도 | 복잡도 |
|----------|---------|--------|--------|
| 데이터 모델 | 유아 발달 도메인 스키마 부재 | 높음 | L |
| UX | 부모 친화적 입력 UI 부재 | 높음 | M |
| 시각화 | 발달 곡선/차트 부재 | 높음 | M |
| AI | 발달 인사이트 엔진 부재 | 중간 | XL |
| 보안 | 유아 데이터 보호 체계 미비 | 높음 | L |
| 공유 | 부모↔교사 데이터 공유 미구현 | 중간 | M |

## 축 5: 경쟁 환경 분석

### 5.1 직접 경쟁
| 서비스명 | 핵심 기능 | 강점 | 약점 |
|----------|-----------|------|------|
| 베이비타임 | 수유/수면/기저귀 기록 | 간편한 입력 | 발달 추적 부재 |
| 키즈노트 | 어린이집↔부모 알림장 | B2B 기관 연동 | 발달 데이터 분석 없음 |
| BabyTracker | 일상 기록 (수유/수면) | 글로벌 사용자 | 한국 로컬라이징 부족 |
| 그로잉 | 성장 기록 + 포토북 | 감성적 UX | AI 분석 없음 |
| 차이의놀이 | 놀이 활동 추천 | 콘텐츠 풍부 | 개인화 발달 추적 없음 |

### 5.2 차별화 방향
- **데이터 중심**: 감성 기록이 아닌 구조화된 발달 데이터
- **시각화**: 발달 영역별 레이더 차트, 성장 곡선, 또래 비교
- **AI 인사이트**: 발달 패턴 분석, 개인화 활동 추천
- **전문가 연결**: 발달 데이터 기반 상담 연계
- **B2B 확장**: 유치원/어린이집 기관용 대시보드

---

## 축 6: 리스크 & 제약 조건

### 6.1 유아 데이터 보호 (최우선)
- **개인정보보호법**: 만 14세 미만 아동 → 법정대리인(부모) 동의 필수
- **COPPA** (해외 확장 시): 13세 미만 아동 데이터 수집 규제
- **민감정보**: 건강 데이터(예방접종, 질병) → 별도 동의 + 암호화
- **데이터 최소화**: 꼭 필요한 데이터만 수집
- **삭제권**: 부모의 데이터 삭제 요청 즉시 처리
- **공유 통제**: 부모가 교사/전문가와의 데이터 공유 범위를 직접 통제

### 6.2 비즈니스 리스크
- 육아 앱의 높은 이탈률 (생활 패턴 변화, 아이 성장)
- 데이터 입력 부담 → 간편 입력 UX가 핵심
- 전문가 검증 없는 AI 판단의 위험성 → 면책 조항 + 전문가 연계 필수

# RULES

1. **KidsMap은 독립 프로젝트** — QETTA와 혼동하지 마라
2. 추측하지 말고 질문하라 — 불확실한 정보는 반드시 질문으로 확인
3. 구체적 수치를 요구하라 — "빠르게" 대신 "FCP 1.2초"
4. 유아 데이터 보호를 최우선하라 — Privacy by Design
5. 금지 용어: "혁신적", "획기적" → 구체적 수치와 개선 폭으로 표현
6. Plan Mode 진입 전까지는 실행 제안 금지 — 탐색과 정리에만 집중
7. 디자인 리소스(Catalyst, Compass) 활용을 우선하라 — 처음부터 만들지 말 것
```

## 5. KidsMap 발달 영역 구조

유아 발달 데이터의 구조화 기준:

| 영역 ID | 이름 | 추적 항목 | 데이터 유형 |
|----------|------|-----------|-------------|
| PHYSICAL | 신체 발달 | 키, 몸무게, 대근육, 소근육 | 수치, 체크리스트 |
| COGNITIVE | 인지 발달 | 수 개념, 분류, 패턴 인식, 주의 집중 | 체크리스트, 관찰 메모 |
| LANGUAGE | 언어 발달 | 어휘량, 문장 길이, 발음, 이해력 | 수치, 텍스트, 음성 |
| SOCIAL | 사회성 | 또래 관계, 규칙 이해, 협동, 나눔 | 관찰 체크리스트 |
| EMOTIONAL | 정서 | 감정 표현, 조절, 자존감, 분리 불안 | 관찰 체크리스트, 메모 |
| HABITS | 생활 습관 | 수면, 식사, 배변, 자조 기술 | 수치, 시간, 체크리스트 |
| HEALTH | 건강 | 예방접종, 알레르기, 질병, 신체검사 | 날짜, 텍스트, 파일 |

### 연령별 발달 기준표 (한국 표준)

```
0~12개월: 영아 발달 (감각운동기)
13~24개월: 걸음마기 (탐색·자율성)
25~36개월: 유아 초기 (언어 폭발기)
37~48개월: 유아 중기 (사회성 발달)
49~60개월: 유아 후기 (학교 준비기)
61~84개월: 취학 전 (논리적 사고 시작)
```
